# Essential Components of Tables (Relational Model)

A **table** (also called a **relation**) is the core structure used to store data in a **relational database**. Understanding its components is fundamental for learning **DBMS, SQL, data modeling, and data analytics**.

This document explains **rows (tuples), columns (attributes), cardinality, degree, constraints, and keys** in **simple language with clear examples**.

---

## 1. Table (Relation)

A **table** is a collection of data organized in **rows and columns**.

### Example: `STUDENT` Table

| Student_ID | Name  | Age | Department |
| ---------: | ----- | --: | ---------- |
|        101 | Rahul |  20 | CSE        |
|        102 | Asha  |  21 | ECE        |
|        103 | Karan |  19 | ME         |

---

## 2. Row (Tuple)

A **row** represents **one complete record** in a table.

* Each row contains data related to **one entity**
* Also called a **tuple** in relational databases

### Example

| Student_ID | Name  | Age | Department |
| ---------: | ----- | --: | ---------- |
|    **101** | Rahul |  20 | CSE        |

This row represents **one student** → Rahul.

### Key Points

* Horizontal in nature
* One row = one real-world object

---

## 3. Column (Attribute)

A **column** represents a **property or characteristic** of an entity.

### Example Columns in `STUDENT`

* `Student_ID` → Identifies student
* `Name` → Student name
* `Age` → Student age
* `Department` → Student department

### Column Example

| **Name** |
| -------- |
| Rahul    |
| Asha     |
| Karan    |

### Key Points

* Vertical in nature
* All values in a column are of the **same data type**

---

## 4. Cardinality

**Cardinality** refers to the **number of rows (tuples)** in a table.

### Example

`STUDENT` table has:

* 3 rows

➡ **Cardinality = 3**

### Formula

```
Cardinality = Total number of rows
```

### Why It Matters

* Indicates **table size**
* Important for **performance and storage analysis**

---

## 5. Degree

**Degree** refers to the **number of columns (attributes)** in a table.

### Example

`STUDENT` table columns:

1. Student_ID
2. Name
3. Age
4. Department

➡ **Degree = 4**

### Formula

```
Degree = Total number of columns
```

### Why It Matters

* Shows how **wide** a table is
* Important during **database normalization**

---

## 6. Constraints

**Constraints** are **rules** applied to table columns to ensure **data accuracy, consistency, and integrity**.

### Common Types of Constraints

| Constraint  | Purpose                      |
| ----------- | ---------------------------- |
| NOT NULL    | Prevents empty values        |
| UNIQUE      | Ensures no duplicate values  |
| PRIMARY KEY | Uniquely identifies each row |
| FOREIGN KEY | Links two tables             |
| CHECK       | Validates values             |
| DEFAULT     | Assigns default value        |

---

### 6.1 NOT NULL Constraint

Ensures a column **cannot be empty**.

```sql
Name VARCHAR(50) NOT NULL
```

❌ Invalid:

```
Name = NULL
```

---

### 6.2 UNIQUE Constraint

Ensures **no duplicate values** in a column.

```sql
Email VARCHAR(100) UNIQUE
```

---

### 6.3 CHECK Constraint

Ensures values satisfy a **condition**.

```sql
Age INT CHECK (Age >= 18)
```

---

### 6.4 DEFAULT Constraint

Assigns a **default value** if none is provided.

```sql
Country VARCHAR(20) DEFAULT 'India'
```

---

## 7. Keys

**Keys** are special columns used to **identify rows uniquely** and **create relationships** between tables.

---

### 7.1 Primary Key

A **Primary Key**:

* Uniquely identifies each row
* Cannot be NULL
* Cannot have duplicates

### Example

```sql
Student_ID INT PRIMARY KEY
```

| Student_ID | Name  |
| ---------: | ----- |
|        101 | Rahul |
|        102 | Asha  |

---

### 7.2 Candidate Key

* Columns that **can qualify** as a primary key

### Example

| Student_ID | Email                             |
| ---------: | --------------------------------- |
|        101 | [r@gmail.com](mailto:r@gmail.com) |

Candidate Keys:

* Student_ID
* Email

---

### 7.3 Alternate Key

* Candidate keys **not chosen** as primary key

If:

* `Student_ID` = Primary Key

➡ `Email` becomes **Alternate Key**

---

### 7.4 Composite Key

* A key made using **two or more columns**

### Example

```sql
PRIMARY KEY (Student_ID, Course_ID)
```

Used when **one column alone is not sufficient**.

---

### 7.5 Foreign Key

* A column that **references primary key of another table**
* Maintains **referential integrity**

### Example

**STUDENT**

```sql
Student_ID INT PRIMARY KEY
```

**ENROLLMENT**

```sql
Student_ID INT FOREIGN KEY REFERENCES STUDENT(Student_ID)
```
