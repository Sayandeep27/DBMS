# Relational Model in DBMS

---

## üìå What is the Relational Model?

The **Relational Model** is a way of organizing data in the form of **tables** (also called **relations**).
Each table consists of **rows** and **columns**, where:

* **Rows** represent individual records
* **Columns** represent attributes (properties)

This model was proposed by **E. F. Codd** and is the foundation of most modern databases like **MySQL, PostgreSQL, Oracle, SQL Server**, etc.

* Table Name: **STUDENT**

## üìò Important Terms in Relational Model

### 1Ô∏è‚É£ Table (Relation)

A **table** is a collection of related data stored in rows and columns.

**Example:** STUDENT table

---

### 2Ô∏è‚É£ Row (Tuple)

A **row** represents a single record in a table.

**Example:**

```
(101, Rahul, 20, B.Tech)
```

This row represents one student.

---

### 3Ô∏è‚É£ Column (Attribute)

A **column** represents a property or characteristic of an entity.

**Examples:** Roll_No, Name, Age, Course

---

### 4Ô∏è‚É£ Record

A **record** means one complete row in a table.

‚úî Every **tuple is a record**
‚úî Every **record represents one real-world entity**

---

### 5Ô∏è‚É£ Domain

A **domain** defines the **type of values** an attribute can store.

| Attribute | Domain Example     |
| --------- | ------------------ |
| Roll_No   | Integer (positive) |
| Name      | String (text)      |
| Age       | Integer (0‚Äì100)    |
| Course    | String             |

---

### 6Ô∏è‚É£ Degree

The **degree** of a relation is the **number of columns** in a table.

**Example:**

* STUDENT table has 4 columns

```
Degree = 4
```

---

### 7Ô∏è‚É£ Cardinality

The **cardinality** of a relation is the **number of rows (tuples)** in a table.

**Example:**

* STUDENT table has 3 rows

```
Cardinality = 3
```

---

## üß© Relation Schema

A **relation schema** describes the structure of a table.

**Syntax:**

```
Relation_Name(Attribute1, Attribute2, Attribute3, ...)
```

**Example:**

```
STUDENT(Roll_No, Name, Age, Course)
```

## üîë Keys in Relational Model (Overview)

| Key Type      | Meaning                                             |
| ------------- | --------------------------------------------------- |
| Primary Key   | Uniquely identifies a row                           |
| Candidate Key | Possible primary keys                               |
| Foreign Key   | Refers to primary key of another table              |
| Composite Key | Key made of multiple attributes                     |
| Super Key     | Any set of attributes that uniquely identifies rows |

## ‚úÖ Advantages of Relational Model

* Simple and easy to understand
* Data is stored in structured form
* Supports powerful SQL queries
* Ensures data integrity using constraints
* Easy to maintain and modify

---

## ‚ùå Limitations of Relational Model

* Not ideal for very complex data (like graphs)
* Performance issues with very large joins
* Less flexible for unstructured data

---

## üß† Real-Life Example

Think of a **college register**:

* Each page = Table
* Each student entry = Row
* Each column = Attribute

That is exactly how the **Relational Model** works.

---

## üìå Summary

* Relational Model stores data in **tables**
* Rows = **Tuples**, Columns = **Attributes**
* Degree = number of columns
* Cardinality = number of rows
* Domains restrict valid values
* Keys help uniquely identify data
